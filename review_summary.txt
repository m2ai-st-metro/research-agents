1. WHAT THIS IS
═══════════════════════════════════════════════════════════════════════

A Karpathy-style closed-loop experiment system that auto-optimizes the search
queries used by production research agents (arxiv, tool_monitor, domain_watch,
youtube, rss). It mutates queries using local Ollama (qwen2.5:14b), runs them
through a mini-pipeline, scores results via Non-Dismiss Rate (NDR), validates
winners with Claude Sonnet, and auto-commits winning mutations to config.py.

Entry point:  ~/projects/research-agents/run-experiments.py
Ledger DB:    ~/projects/research-agents/auto_research/data/experiments.db (52 KB)

5. BLOCKING ISSUES & DECISIONS NEEDED
═══════════════════════════════════════════════════════════════════════

ISSUE 1 — insufficient_data dominates (32 of 79 runs = 40%)
  What: Experiments fail because fewer than 5 relevant signals are returned
        (MIN_SIGNALS_PER_EXPERIMENT = 5 in auto_research/config.py).
  Impact: Only 52% of experiments produce usable comparisons.
  Root cause: Many query combinations don't reliably return 5+ scorable signals
              from ArXiv + GitHub + HN in the mini-pipeline.
  Decision needed: Lower MIN_SIGNALS_PER_EXPERIMENT to 3? Or widen search
                   sources in mini_pipeline.py? Or switch to a different
                   baseline (signal count vs NDR) for sparse queries?

ISSUE 2 — RSS feed mutations produce invalid_config (5 experiments)
  What: The mutator generates free-form text descriptions ("AI industry insights")
        when asked to mutate RSS feed names (e.g. "The Batch", "Latent Space").
        These aren't valid RSS feed identifiers and can't be scored.
  Impact: RSS agent can't benefit from the experiment loop.
  Decision needed: Either exclude rss from EXPERIMENT_AGENTS or teach the
                   mutator to generate valid RSS feed URLs/names rather than
                   descriptive text. File: auto_research/mutator.py ~line 177.

ISSUE 3 — YouTube agent: 0 winners from 10 runs
  What: youtube has the highest avg variant NDR (0.87) but no wins vs baseline.
  Impact: YouTube queries are not improving despite active mutation.
  Likely cause: Baseline queries may already be well-optimized, or the mini-
                pipeline doesn't search YouTube (only arXiv, GitHub, HN) so
                YouTube variants can't be fairly evaluated in simulation.
  Decision needed: Is YouTube even evaluable in the mini-pipeline? If not,
                   remove it from EXPERIMENT_AGENTS (currently included per
                   auto_research/config.py).

ISSUE 4 — Rollback check not integrated
  What: `python run-experiments.py --rollback-check` prints:
        "Rollback check requires production NDR — not yet integrated"
  Impact: Weekly regression detection is non-functional. If a committed
          mutation degrades real pipeline quality, it won't be caught.
  Decision needed: Implement production NDR tracking by reading IdeaForge
                   DB signal scores as the ground truth. See auto_research/
                   committer.py: check_weekly_rollback().

ISSUE 5 — No weekly baselines have ever been saved
  What: weekly_baselines table in experiments.db is empty.
  Impact: Rollback check can never work without a baseline to compare against.
  Decision needed: Run `ledger.save_weekly_baseline()` on a schedule (or hook
                   it into the cron job). Needed before rollback check is useful.

ISSUE 6 — Guardrail mismatch between simulation and production
  What: 1 experiment was guardrail_blocked (NDR up but avg_score dropped).
        The arxiv "contextual model protocol analysis" query shows conflicting
        results — guardrail_blocked in one run, completed winner in the next.
        This suggests Ollama's scoring is non-deterministic enough to produce
        opposite verdicts for the same query pair.
  Impact: Winners may be promoted that wouldn't survive Claude validation;
          Claude validation catches some of this but not all.
  Decision needed: Add deterministic seed or multiple-run averaging to reduce
                   scoring variance before promoting a winner.
